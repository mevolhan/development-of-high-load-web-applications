# Лабораторные работы по высоконагруженным системам

## Лабораторная работа №1: Балансировка и отказоустойчивость с Nginx

**Описание:**  
Разработано Java-приложение на Spring Boot с эндпоинтом `/api/counter`, возвращающим увеличивающийся счётчик. Три экземпляра приложения запущены в Docker, Nginx в роли балансировщика нагрузки.

**Технологии:**  
Java 17, Spring Boot, Docker, Nginx, JMeter

### Алгоритмы балансировки
- Round Robin (по умолчанию)
- Least Connections (`least_conn`)
- IP Hash (`ip_hash`)
- Random (`random`)

### Отказоустойчивость
Настроены параметры `max_fails=2` и `fail_timeout=30s` для автоматического исключения отказавшего сервера и его восстановления.

### Тестирование
Нагрузочное тестирование проведено с помощью Apache JMeter.

## Лабораторная работа №2: Разделяемое хранилище данных с Redis

## Описание
Разработано Java-приложение на Spring Boot с двумя функциональными модулями:
- **Счётчик запросов** — общее значение хранится в Redis и увеличивается при каждом обращении к `/api/counter`.
- **Таблица лидеров (Leaderboard)** — на основе Sorted Set Redis реализован REST API для добавления очков игроков и получения топа.

Три экземпляра приложения запущены в Docker, Nginx выступает в роли балансировщика нагрузки, Redis — единое хранилище состояния.

## Технологии
- Java 17, Spring Boot, Spring Data Redis
- Docker, Docker Compose
- Nginx
- Redis

## Реализация
- **Redis** используется как разделяемое хранилище для счётчика и лидерборда.
- **Sorted Set** обеспечивает автоматическую сортировку игроков по очкам.
- **REST API**:
  - `POST /leaderboard/{playerId}?score=` — добавить/обновить очки.
  - `GET /leaderboard/top?limit=` — получить топ N игроков.
  - `GET /leaderboard/rank/{playerId}` — узнать ранг игрока.
- **Docker Compose** поднимает три экземпляра приложения, Redis и Nginx в единой сети.

## Проверка работы
- Счётчик инкрементируется независимо от того, на какой экземпляр попал запрос.
- Данные лидерборда сохраняются в Redis и доступны всем экземплярам.
- Балансировка Nginx распределяет запросы между тремя контейнерами.

## Лабораторная работа №3: Реализация очередей задач с использованием Redis

### Описание  
Реализована асинхронная система обработки задач на основе Redis. Генератор с неравномерной задержкой (0–5 с) создаёт задачи и помещает их в очередь. Обработчик извлекает задачи, преобразует текст в верхний регистр и публикует результат в канал. REST API принимает пользовательские запросы, отправляет задачу в очередь и возвращает ответ клиенту через канал.

### Технологии  
Java 17, Spring Boot 3.2.5, Spring WebFlux, Spring Data Redis Reactive, Redis, Docker, Lombok

### Ключевые особенности  
- **Неравномерная генерация задач** – случайные задержки от 0 до 5 секунд.  
- **Реактивное взаимодействие с Redis** – использование `ReactiveRedisTemplate`.  
- **Асинхронная обработка** – задачи обрабатываются независимо от приёма запросов.  
- **Docker‑изоляция** – собственные контейнеры Redis и приложения, не пересекаются с предыдущими работами.  
- **Дополнительные эндпоинты** – `/api/tasks` (список очереди) и `/api/tasks/count` (длина очереди) для мониторинга.  

### Демонстрация  
- POST `/api/tasks` с телом `{"data":"hello"}` возвращает `{"result":"Processed: HELLO"}`.  
- В логах видна автоматическая генерация и обработка задач с разными интервалами.  
- GET-запросы показывают текущее состояние очереди.

## Лабораторная работа №5: Партиционирование в PostgreSQL

**Описание:**  
Реализовано декларативное партиционирование таблицы `orders` по диапазону дат. Созданы партиции для января, февраля и марта 2024 года. Выполнены вставка данных, добавление и удаление партиций, создание локального и составного индексов. С помощью `EXPLAIN ANALYZE` продемонстрированы использование индексов и отсечение партиций.

**Технологии:**  
PostgreSQL 15, Docker, pgAdmin 4, SQL

**Операции с партициями:**  
- Создание партиционированной таблицы `orders` с разбиением по `order_date`
- Добавление партиций (`CREATE TABLE ... PARTITION OF`)
- Удаление партиции (`DROP TABLE`)
- Отсоединение партиции (`ALTER TABLE ... DETACH PARTITION`)

**Индексы:**  
- Локальный индекс по полю `order_date`
- Составной индекс по полям `(customer_id, order_date)`

**Анализ производительности:**  
- Использование `EXPLAIN ANALYZE` для проверки применения индексов
- Демонстрация отсечения партиций (partition pruning) при фильтрации по ключу партиционирования
